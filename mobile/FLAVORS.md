# Build flavors — Xarid (customer) + Xarid Kuryer (courier)

One Flutter codebase ships as **two** Android/iOS apps that can be installed
side by side. The Dart is identical; only the native app id, display name and
launcher icon differ. The signed-in user's role still decides which UI loads at
runtime (see `lib/main.dart` `_Gate`) — the flavor only sets the window title
and lets the courier build default-hint the courier UI when the role is unknown.

| Flavor     | App name      | Android applicationId | iOS bundle id        | Default entry            |
|------------|---------------|-----------------------|----------------------|--------------------------|
| `customer` | Xarid         | `uz.xarid.app`        | `uz.xarid.app`       | `lib/main_customer.dart` |
| `courier`  | Xarid Kuryer  | `uz.xarid.courier`    | `uz.xarid.courier`   | `lib/main_courier.dart`  |

> **Why this file exists:** `android/` and `ios/` are generated by
> `flutter create` and are **gitignored** in this repo, so the native flavor
> wiring below is not committed. Re-apply these steps after regenerating the
> platform folders.

## Dart side (committed, already done)

- `lib/main.dart` — shared bootstrap. Exposes `runXarid()` and a compile-time
  flag `const appFlavor = String.fromEnvironment('FLAVOR', defaultValue: 'customer')`.
  The flag only changes the `MaterialApp.title` (and the courier default-hint).
  Plain `flutter run` (default entry `lib/main.dart`) still boots the customer app.
- `lib/main_customer.dart`, `lib/main_courier.dart` — thin entry points that
  call `runXarid()`.

Pass the flavor at build time with `--dart-define=FLAVOR=...` so the Dart title
matches the native flavor (the two are independent — keep them in sync).

## Run / build commands

```sh
# Customer ("Xarid")
flutter run            --flavor customer -t lib/main_customer.dart --dart-define=FLAVOR=customer
flutter build appbundle --flavor customer -t lib/main_customer.dart --dart-define=FLAVOR=customer
flutter build apk       --flavor customer -t lib/main_customer.dart --dart-define=FLAVOR=customer
flutter build ipa       --flavor customer -t lib/main_customer.dart --dart-define=FLAVOR=customer

# Courier ("Xarid Kuryer")
flutter run            --flavor courier -t lib/main_courier.dart --dart-define=FLAVOR=courier
flutter build appbundle --flavor courier -t lib/main_courier.dart --dart-define=FLAVOR=courier
flutter build apk       --flavor courier -t lib/main_courier.dart --dart-define=FLAVOR=courier
flutter build ipa       --flavor courier -t lib/main_courier.dart --dart-define=FLAVOR=courier
```

The existing `--dart-define=API_BASE_URL=https://...` override (see `lib/config.dart`)
can be appended to any of these.

## Android

This project uses the **Kotlin DSL** (`android/app/build.gradle.kts`) and its
`namespace`/`applicationId` are already `uz.xarid.app`.

### 1. Declare the product flavors

In `android/app/build.gradle.kts`, inside the `android { }` block, add a flavor
dimension and the two flavors. `resValue` injects a per-flavor `app_name` string;
`applicationIdSuffix` keeps the courier build installable alongside the customer.

```kotlin
android {
    // ...existing config...

    flavorDimensions += "app"
    productFlavors {
        create("customer") {
            dimension = "app"
            // applicationId stays uz.xarid.app (the defaultConfig value)
            resValue(type = "string", name = "app_name", value = "Xarid")
        }
        create("courier") {
            dimension = "app"
            applicationIdSuffix = ".courier"   // -> uz.xarid.courier
            resValue(type = "string", name = "app_name", value = "Xarid Kuryer")
        }
    }
}
```

> `applicationIdSuffix = ".courier"` turns the base `uz.xarid.app` into
> `uz.xarid.app.courier`. If you need exactly `uz.xarid.courier`, set the full id
> instead: replace the suffix line with `applicationId = "uz.xarid.courier"`.

### 2. Use the injected name in the manifest

`android/app/src/main/AndroidManifest.xml` currently hardcodes
`android:label="xarid"`. Change it to the generated string resource:

```xml
<application
    android:label="@string/app_name"
    android:name="${applicationName}"
    android:icon="@mipmap/ic_launcher">
```

### 3. Per-flavor source sets (optional, for icons)

Gradle automatically merges resources from `src/<flavor>/` over `src/main/`.
`flutter_launcher_icons` (configured with a `flavor:` key, below) writes the
courier mipmaps into `android/app/src/courier/res/`, so no manual copying is needed.

## iOS

Flutter maps an Android flavor to an **Xcode scheme + build configuration** of
the same name. Set this up once in `ios/Runner.xcodeproj` (Xcode):

1. **Build configurations** — duplicate the stock `Debug` / `Release` / `Profile`
   into per-flavor pairs: `Debug-customer`, `Release-customer`, `Profile-customer`
   and `Debug-courier`, `Release-courier`, `Profile-courier`.
2. **Schemes** — create a scheme named exactly `customer` and one named `courier`
   (Product -> Scheme -> Manage Schemes -> +). Mark them **Shared** so they are
   committed. Point each scheme's Run/Archive actions at its matching
   `*-customer` / `*-courier` configurations.
3. **Bundle id per configuration** — in the Runner target's Build Settings set
   `PRODUCT_BUNDLE_IDENTIFIER`:
   - `*-customer` -> `uz.xarid.app`
   - `*-courier`  -> `uz.xarid.courier`
4. **Display name** — set `PRODUCT_NAME` (or `INFOPLIST_KEY_CFBundleDisplayName`)
   per configuration to `Xarid` / `Xarid Kuryer`, or reference a build setting
   from `Info.plist` via `$(BUNDLE_DISPLAY_NAME)`.

Flutter then matches `--flavor customer` to the `customer` scheme automatically.

## Per-flavor icons (flutter_launcher_icons)

`pubspec.yaml` keeps the default `flutter_launcher_icons:` block, which generates
the **customer** icon from `assets/icon.png`. Add a second, flavor-scoped config
file for the courier icon (drop a `assets/icon_courier.png` next to it first):

`mobile/flutter_launcher_icons-courier.yaml`
```yaml
flutter_launcher_icons:
  android: true
  ios: true
  flavor: courier                 # writes Android res into src/courier/, iOS into a courier asset set
  image_path: "assets/icon_courier.png"
  remove_alpha_ios: true
  background_color_ios: "#FFFDF1"
  adaptive_icon_background: "#FFFDF1"
  adaptive_icon_foreground: "assets/icon_courier.png"
```

Generate:

```sh
dart run flutter_launcher_icons                                   # customer (default pubspec block)
dart run flutter_launcher_icons -f flutter_launcher_icons-courier.yaml   # courier
```

For an explicit customer config (instead of the default block) create
`flutter_launcher_icons-customer.yaml` with `flavor: customer` the same way.

## Splash screen

`flutter_native_splash` is shared (cream `#FFFDF1`, `assets/icon.png`) and does
not need per-flavor variants. If a courier-specific splash is wanted later, it
supports the same `flavor:` key via a `flutter_native_splash-courier.yaml` file.
